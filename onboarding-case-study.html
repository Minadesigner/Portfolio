Onboarding as a System

Designing a Scalable Supplier Intake for Enterprise TPRM

Transforming a manual onboarding bottleneck into a scalable, data-controlled system — reducing time-to-value and enabling enterprise teams to onboard 50–500 suppliers efficiently.

The Problem

Onboarding built for one supplier at a time, in a system that needed to handle hundreds

Supplier onboarding is the entry point to all TPRM workflows — compliance, audits, and risk monitoring. Yet the experience was designed around manual, one-by-one data entry.

For enterprise customers managing 50–500 suppliers, this didn't scale — turning onboarding into a bottleneck rather than a starting point.

01
Manual entry at scale
Session recordings showed users hesitating on field requirements, scrolling repeatedly, and spending significantly more time than the task warranted.
02
No bulk operations
Teams had no way to import existing vendor data. The workaround was offline spreadsheets and backend engineering imports.
03
Unclear field requirements
No distinction between required and optional fields — users hesitated, skipped fields, and had to go back to complete records.
04
Downstream data risk
Incorrect data propagated silently into compliance and audit workflows, reducing trust in the system.

The original form — one-by-one entry, no field hierarchy, no bulk support.

Research & Discovery

Why users abandoned the onboarding flow entirely

Observing users on the platform and gathering feedback from the sales team revealed two separate but connected problems — one about speed, one about accuracy.

01 — Key Insight
The form exposed users to too much at once
Source: Session recordings (Clarity)
Design Decision
Show only what matters first — start with the supplier name
Users saw too many fields immediately, many not mandatory. The new flow starts with just the supplier name, auto-suggests from a database, and finds the domain automatically — reducing decision load from the first step.
02 — Key Insight
Speed alone wasn't enough — accuracy was the real risk
Source: Session recordings · observed incorrect supplier and domain entries
Design Decision
Validate at every step, not just at submission
Users were adding wrong supplier names and incorrect domains. The new flow validates domains automatically, pre-fills data from an API, and lets users modify — ensuring accuracy without adding friction.
03 — Key Insight
Enterprise clients were already working in spreadsheets
Source: Sales team feedback
Design Decision
Meet users where they are — accept CSV and Excel uploads
Clients managing 50–500 suppliers had their data in spreadsheets already. Rather than asking them to change their workflow, we built an import flow that accepts their existing files — with guided mapping to ensure data lands correctly.
04 — Key Insight
Bulk import without validation creates a bigger problem
Source: Team discussion · observed data accuracy issues
Design Decision
Gate import on mandatory fields — surface errors before they enter the system
Speed of import is worthless if the data is wrong. The flow requires mandatory fields to be mapped, validates domains, and flags errors before import — users must fix issues first.
Design Approach

From manual input to scalable onboarding

I reframed onboarding as a system that needs to support both scale and flexibility. Rather than optimizing a single flow, the goal was to align the experience with how users actually manage supplier data.

Solution 01
Bulk import

For enterprise teams already managing suppliers in spreadsheets. Upload a CSV, map columns to system fields, validate before committing — controlled onboarding at scale.

Solution 02
Simplified single-supplier flow

For everyday use. Start with a name, get auto-suggestions, validate the domain automatically. Only the essential fields — with smart defaults and the option to skip the rest.

Bulk import is not just about speed — it is controlled onboarding at scale.

The Solution

A flexible onboarding system

The final design introduces two aligned entry points — both sharing the same data model and validation logic, ensuring consistency across the platform.

Key design decisions
Auto-mapping fields
Normalizes column names and pre-matches fields to reduce setup time.
Required field gating
Users cannot continue without mapping name and domain — the two critical fields.
Validation before creation
Incorrect data does not enter the system — errors must be fixed first.
Re-upload instead of inline editing
CSV errors are usually systemic — re-uploading the fixed file is faster than row-by-row editing.

In enterprise onboarding, the biggest risk is not uploading data — it is importing incorrect data.

02
Guided form

Simplified single-supplier flow

Bulk import solves scale, but users still need to add suppliers individually. The manual flow was redesigned to reduce friction and ambiguity.

What changed
Required vs optional fields — clearly labelled
Users immediately see what they must fill and what they can skip.
Structured sections
Company, risk context, and monitoring grouped separately — reducing cognitive load.
Preselected defaults
API pre-fills data where available — users review and adjust instead of typing from scratch.
Improved visual hierarchy
Faster to scan, faster to complete — without sacrificing flexibility.
Outcome

Operations teams stopped worrying about onboarding

The redesigned flow eliminated the workarounds entirely. Teams now onboard hundreds of suppliers in a single session — without spreadsheet exports, backend engineering, or data correction tickets.
